## 正文

要理解 Eventloop，首先我們需要先理解為什麼會需要 Eventloop。這對於理解該工具的角色定位是很重要的。

眾所週知，Javascript 是一個單執行緒 (Single-threaded) 的語言，單執行緒的意思是，他是單線程的，1 次只能做 1 件事情。更深入點的探討，他只有一個 **callback stack**。

想像一條流水線，有個工人站在流水線上拆封包裹，每個包裹拆封的複雜度不一樣，有些很容易處理，有些則需要耗時很長時間。如果一個包裹需要耗時很長時間，這個工人卻堅持將包裹處理完再去處理下一個，這將導致流程堵塞，也就是 **blocking**。

因此，這位工人並不會停下整條流水線死守著它。相反地，他會把這個包裹貼上一張標籤，隨手丟給旁邊的**自動化機器（瀏覽器 Web APIs）**去處理。然後，工人會立刻轉身處理流水線上的下一個包裹，完全不耽誤進度。

等到那台自動化機器處理完那個複雜包裹後，它會把處理好的結果輸送到一個 **『待辦籃子』（Callback Queue） **裡排隊。

我們的主角：一位眼神銳利的 **監控員Event Loop** 出來了。
這位監控員永遠只盯著兩件事：工人的手，以及那個籃子。

工人非常有原則，只要流水線上還有眼前的包裹，他就不會分心。而監控員也非常有耐心，他會一直站在旁邊觀察，直到發現工人手頭上的工作全都清空了、拍拍灰塵準備休息的那一秒，監控員才會敏捷地從『待辦籃子』裡抓起排在第一位的成果，精準地拋到工人面前。

這時，工人才會重新動手，依序完成最後的標記或裝箱。

這套**『工人拆封、機器代工、監控員傳球』**的運作流程，就是 JavaScript 雖然只有單執行緒，卻能同時處理很多事情且不卡頓的秘密武器。

這篇故事的一個關鍵點是，所謂的『耗時很長』是誰決定的？

在 JavaScript 的真實世界裡，這些包裹通常被稱為 **非同步請求（Asynchronous Requests）**。所謂的『耗時很長』，通常指的不是計算邏輯太複雜，而是**等待**：

* **等待遠方的回應 (AJAX 請求)：** 前端跟後端服務器發出請求，等待服務器回應，回應的時間是不固定的，所以必定是屬於非同步請求。
* **等待時間抵達 (setTimeout/setInterval)：** 就像設定一個鬧鐘，工人沒必要盯著秒針轉動，他會把鬧鐘交給瀏覽器的定時器。
* **等待使用者的動作 (DOM Events)：** 就像守候一個不知道何時會開的門，工人會把「有人開門就通知我」的標籤貼給瀏覽器的事件監聽器，比如點擊事件。

### 真正的分工：JavaScript 引擎 vs 瀏覽器環境

這裡有一個常見的誤解：很多人以為 `setTimeout` 或 `fetch` 是 JavaScript 語言的一部分。

事實上並不是。

JavaScript 引擎 (工人) 本身非常純粹，它只負責執行程式碼。而那些自動化機器 (Web APIs) 則是瀏覽器提供的額外工具。這就是為什麼同樣的 JavaScript 程式碼，在瀏覽器執行時有 `window` 物件，在 Node.js 執行時則有 `fs` 模組——因為它們連接的是不同的「自動化機器」。

### 監控員（Event Loop）的鐵律：單向循環

監控員（Event Loop）的運作有一個絕對的鐵律：**他永遠不會在工人忙碌時插單。**

這解釋了為什麼當寫下 `setTimeout(callback, 0)` 時，該任務並不會「立刻」執行。即使機器處理時間是 0 秒，它依然得先去「待辦籃子」裡排隊。監控員必須等到流水線上的包裹清空，才會放行。

這也說明了為什麼當 JavaScript 正在跑一個幾百萬次的死循環運算時，無論你怎麼點擊網頁上的按鈕（觸發新的包裹），網頁都毫無反應。因為監控員看著那個「還在忙碌的工人」，他無法把籃子裡的點擊事件交給他。

## 結語

理解了這套機制，你就會明白為什麼「不阻塞（Non-blocking）」是 JavaScript 的核心靈魂。我們利用了瀏覽器的多執行緒能力（自動化機器），來補足 JavaScript 單執行緒（工人）的限制。
