### 1. 為什麼前端工程師需要理解 UI/UX？
前端工程師並非總是有充足的UI/UX資源的。在沒有Figma，或企劃只有文字，甚至可能只有一個模糊的需求時，前端如果想要完成網頁的設計，必須掌握「產品構建邏輯」。具體來說，掌握產品建構邏輯可以為我們帶來以下幾個好處：

**統一視覺的邏輯，避免無止盡的修改：**
UI一個很重要的原則是，其外觀與畫面元素需具備「統一性」，當建立一套客戶通過的Design System，比如統一的字級，網站的主題配色，卡片清單的Grid等等，就可以依照該規範進行開發，當客戶隨興的提出UI改稿修改時，可以用「一致性」和「資訊層級」為理由說服對方，避免陷入調整地獄。

舉例來說，如果沒有特殊需求，字級可以固定 `H1`, `H2`, `H3`, 內文, 細部說明分別使用Tailwind幫我們制定好的 `text-3xl`, `text-2xl`, `text-xl`, `text-lg`, `text-base`，因為Tailwind制定的這套規則符合人類對於字體的審美心理需求，這也是筆者很喜歡使用的做法。

**彌補美感缺失帶來的信任危機：**
一個很現實的事實是，我們評判一個網站是否好用，願不願意去使用，有很大程度取決於該網站是否美觀。如果一個網站的色調不統一，元件不對齊，畫面簡陋而雜雜亂，將一堆功能塞在一起，很大可能使用者會直接拒絕使用該網站。

因此，理解UI的原則，諸如相似性，接近性，對稱性，連續性，封閉性，圖地原理等，前端工程師需要對其有了解，我們會在下文做更具體的探討。

**解決邏輯缺失的斷層：**
即使UI/UX設計師提供完備的企劃與Figma，也不代表前端工程師就能高枕無憂，因為企劃或者說設計稿是靜態的，設計者只能用類似連環畫與配套文字，箭頭指向等方式闡述設計，而網頁則是一個可以由程式驅動的動態頁面。換言之，很有可能在使用者操作這一塊，設計圖上存在空缺的，一個具備UX思維的前端，需要以自己的專業性彌補該段空缺。

以筆者的工作經驗來說，UX的缺失是常態，舉例來說，設計師通常不會將設計圖的刻劃具體到點擊某按鈕，例如發送AJAX請求以後，如果資料為空時要顯示什麼畫面，失敗時要提示訊息內容是什麼，要以怎樣的形式(比如toast, 彈窗之類的)呈現等。所以開發時，前端工程師能主動補完邊界情況(Edge Case)，將防呆與反饋加進去，這能減少測試階段被退回的機率與次數。

### 2. UI / UX 是什麼？（工程師角度的理解）



* **UI**：這裡我以一個前端工程師的角度闡述，在我眼中，一個頁面是可以被拆解為以下UI元素
    * **視覺邏輯**：也就是一個網站的主題，其主題包含色調(`Primary color`, `Secondary color`)、字體(前文提到的 `xl`, `lg`, `base`, 粗細, 間距等)、`Padding` 與 `margin`、各種螢幕RWD斷點等全局的元素。通常會寫在 `global.css` 或者 `plugins` 裡面做一次性統一的設定(之類要調整也很簡單)。
    * **布局邏輯**：這裡筆者將網頁分成大排版(`Layout`)與小布局(`Grid`)，大排版是決定一個網站的整體布局，以一般前台網站來說，往往只有 `Header`, `Main`, `Footer` 且三者垂直排列，相當簡單。後台則通常是 `Header`，`Sidebar`, `Main`，很少有 `Footer`。小布局則是專注在 `Main` 裡面根據各頁面需要展示的資訊與操作邏輯，使用 `Flex` 與 `Grid`，對齊與留白做更具體的設計。
    * **元件邏輯**：這是網頁排版裡面的具體元件元素，比如表單元素，選單元素與按鈕元素，這部分最適合做成共用UI元件，因為網站需要統一性，換言之，只需要作出一套元件邏輯就可以在整個網頁中做共用，可以節省大量時間同時使網站統一而美觀。例如按鈕元素，不單純只是個點擊事件，光它就有 `default`, `hover`, `active`, `disabled`, `loading` 等狀態，針對狀態做客製化，做成共用元件，才能讓UI顯得統一而富有邏輯。

### 3. 從技術角度如何實現好的 UX？（核心）

前面談到了 UI，接下來聊聊 UX。UX，全名為 User Experience，也就是使用者體驗。筆者認為，在需要高度互動的網頁中，UX 的重要性無庸置疑，甚至往往比 UI 更為關鍵。在開始探討相關理論之前，先分享一段筆者的實際經歷：

筆者曾使用政府某網站，需上傳新冰箱的購買證明、規格書與序號文件，以申請退稅。頁面上清楚說明支援 `jpg`、`jpeg`、`png` 等圖片格式，也可上傳 `doc`、`docx`、`pdf` 等文件。然而，當筆者完成檔案上傳並填寫所有表單後，點擊提交，頁面卻始終顯示「尚未提交文件」的錯誤提示。

筆者反覆嘗試多次，逐一檢查是否有欄位遺漏，甚至將檔案從文件格式改為圖片格式重新上傳，但問題依然無法解決。而從錯誤提示中，只能得知「尚未提交文件」，並沒有提供進一步的指引。

直到最後，筆者偶然發現，在檔案上傳後顯示的縮圖下方，有一行不明顯的淡色文字：「點擊上傳完成」。在點擊該文字後，系統才真正標記文件為已上傳。此時再次提交，流程才順利完成。

看完這篇故事，就可以理解UX對於使用者的重要性，前端工程師設計UX流程時，必須要站在使用者的角度上，設法做出讓使用者能不跳腳、不迷路、不思考就能把事情辦完的流程。



* **使用者路徑(user flow)**: 在本案例中，使用者的操作應該是「上傳文件→提交→完成申請」，然而開發者在流程中加了一個額外步驟「點擊上傳完成」，且沒有被清楚地納入流程指引中，導致使用者在不知情的情況下卡在中途。這反映出 User Flow 設計不完整的問題：流程中不應存在「隱藏步驟」，每一個操作都應該是可預期、可理解的。對前端工程師而言，實作流程時不應只關注功能是否完成，而是要確認整個操作路徑是否連貫、是否符合直覺。
* **資訊的位置(info architecture)**: 在案例中，「點擊上傳完成」這個關鍵按鈕與提示文字被放在縮圖下方，且以淡色文字呈現，導致使用者極難察覺。這是一個典型的資訊層級錯誤：重要資訊被放在低可見性的位置。良好的資訊架構應該讓關鍵操作與提示具備充足的視覺權重。對前端工程師來說，這不只是設計問題，在實作 UI 時也需要主動判斷哪些資訊應該被強調。
* **反饋與容錯(feedback & error handling)**: 當使用者點擊提交時，系統僅回傳「尚未提交文件」，但沒有說明問題出在哪裡，也沒有引導使用者如何修正。良好的反饋機制應該具備明確指出錯誤原因與提供可行的解決方式。對前端工程師而言，錯誤處理不只是顯示錯誤，而是幫助使用者「順利完成任務」。

### 4. UI 的實作不只是「切版」，而是對完形心理學的運用

完形心理學，其中心思想是「整體大於部分組件的總和」，或者說 `2 > 1 + 1`。該理念是在表達，人類的大腦會自動對親密的，相似的，連續的，封閉的組件們同為一體。在設計UI時，我們可以從四個方面對畫面進行設計：



* **親密性**：親密是指距離上的接近，距離接近的元素，大腦會將其自動是為一體。這解釋為何標題與副標題，內文之間的間距應該較小，並且需要與下一組的資訊間隔拉開。
* **相似性**：長得像的東西，大腦會認為功能相同，這解釋為何時常見到各種網站的刪除按鈕被單獨列為紅色，這是要標註的功能的獨特性。
* **連續性**: 人的視線是線性的，例如一篇文章，我們一定習慣它是由上到下，又或者表單畫面，我們習慣看到表單是對齊的，如果表單一上一下，人會感覺相當不舒服。
* **封閉性**: 大腦會補全不完整的形狀，例如資料為空或者還在loading時，設計上常用 `placeholder` 或者灰色長條狀填充畫面，使用者會立刻預期到這部分將來是會有畫面的。

### 5. 常見前端工程師忽略的 UX 問題

在實務開發中，前端工程師往往會將重心放在「功能是否正常運作」，而忽略了「使用者是否順利完成任務」。

1.  **沒有 loading 狀態**：當頁面需要向後端請求資料時，如果沒有任何 loading 提示，使用者很難判斷是「系統還在處理」還是「頁面壞掉了」。實務上，應該至少提供 `spinner` 或 `skeleton`。
2.  **缺乏明確的錯誤提示**：許多系統只會回傳「操作失敗」，但並未說明原因。良好的錯誤處理，應該要能引導使用者修正問題。
3.  **操作沒有回饋**：當使用者點擊按鈕後，如果畫面沒有任何變化，會讓人產生不確定感。常見做法會加入 `loading` 狀態、按鈕 `disabled` 或狀態變化。
4.  **過晚的表單驗證**：如果表單驗證只在「提交時」才觸發，使用者會增加挫折感。較好的做法是即時驗證（例如 `blur` 或 `debounce`）。

### 6. 如何與設計師合作（實務面）

最終的使用者體驗，是設計與工程共同作用的結果。

* **前端開發者也需要去理解「為什麼這樣設計」**：理解設計意圖，才能在實作時做出更合理的判斷。
* **主動補齊設計稿未涵蓋的情境（Edge Case）**：設計稿本質上是靜態的。前端工程師若能主動補齊資料為空、請求失敗、`loading` 狀態等細節，不僅能提升體驗，也能減少溝通成本。
* **在必要時適度 challenge 設計**：並非所有設計都是最優解。若技術上會造成效能問題或邏輯不正確，應提出討論。
* **建立共用語言與規範**：例如元件命名、狀態定義、Design System 規範等，一致的理解可以提高開發效率。

### 7. 總結
回到文章一開始的問題：為什麼前端工程師需要理解 UI/UX？答案其實已經貫穿全文——因為前端工程師並不只是「把畫面做出來的人」，而是實際決定使用者體驗的人。

從 UI 的角度來看，我們不只是切版，而是在建立一套具有一致性與邏輯的系統；從 UX 的角度來看，我們不只是讓功能能用，而是讓使用者能「不思考地完成任務」。無論是補齊設計稿未涵蓋的邊界情境、優化使用者路徑，還是透過完形心理學去引導使用者理解畫面，本質上都是在降低使用者的認知成本。

最後，UI/UX 並不是設計師的專屬領域，而是一個需要設計與工程共同完成的過程。當前端工程師能理解設計意圖、補足設計缺口，甚至在必要時提出優化建議時，產品的品質才會真正提升。
