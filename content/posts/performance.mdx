## 前言
提到效能優化，一般來說想到的都是 SQL 語法的優化、資料庫索引、演算法的優化或是服務器硬體升級，但是這些是後端與運維工程師的範疇，因此不在這篇文章討論的範圍內。

效能是業界在軟體工程上非常重視的點，不論是 Web 還是 App，都會想追求極致的效能，但是效能優點具體能帶來什麼好處？前端具體應該能做些什麼？這是本文想要討論的。

## 效能優點具體能帶來什麼好處？
* SEO
* 使用者體驗(UX)
* 影響營收的重要指標

### SEO: 
全稱 Search Engine Optimization，執行 SEO 是為了讓網站在 Google 搜尋排名上升，提升網站的能見度與流量，進而創造業績。

### 使用者體驗(UX)：
試想一下，你進到一個電商網站，想找「草莓口味分離乳清蛋白粉」，在搜尋框每打一個字，網站就立刻發送一次 API 請求去資料庫撈資料。你打了 12 個字，搜尋框就跳 12 個請求。結果網頁因為瞬間處理太多回傳資料畫面開始卡頓，然後搜尋結果像抽風一樣一格一格跳出來。只是想搜個口味而已，這網站至於嗎？

### 影響營收的重要指標：
根據 Google 的研究，網頁加載時間從 1 秒增加到 3 秒，跳出率會增加 32%；如果增加到 5 秒，跳出率更是直接飆升 90%。使用者是很殘酷的，他們不會等你把那些精美的 CSS 動畫跑完，只要卡個兩秒，他們就會直接關掉分頁去買別家的蛋白粉。

來源: [Google Think - Mobile Page Speed New Industry Benchmarks](https://business.google.com/ca-en/think/marketing-strategies/mobile-page-speed-new-industry-benchmarks/)

## 前端具體應該能做些什麼？
我們可能認為，效能的優化主要應該交給後端工程師，因為資料庫檢索速度、服務器響應時間（TTFB）這些都是後端負責的範疇，前端只是單純負責切版，發送請求與接收回傳資料再渲染到畫面上，但事實上，前端能做的，要做的，遠比你想得多。

當瀏覽器拿到 HTML/CSS/JS 之後，瀏覽器會執行一套名為 **關鍵渲染路徑 (Critical Rendering Path, CRP)** 的機制。要談前端優化，需要對這顆「樹」是怎麼種出來的有一些了解。



1. **構建 DOM 樹**，瀏覽器解析 HTML 標籤，將其轉化為一個樹狀結構。
2. **構建 CSSOM 樹**，解析 CSS ，它需要知道每個選擇器應用了哪些規則。
3. **將兩棵樹結合為 Render Tree**，這是最關鍵的一步。瀏覽器會把 DOM 與 CSSOM 結合，生成 Render Tree。
4. **算出版面配置**，瀏覽器根據 Render Tree 計算每個節點在螢幕上的幾何位置與大小。這就是 **Reflow(回流)** 發生的地方。
5. **繪製**，最後，瀏覽器將 Render Tree 中的每個節點轉換為螢幕上的實際像素，比如改變顏色、陰影等不重算 Layout 的屬性。這就是 **Repaint (重繪)** 發生的地方。

具備以上認知以後，順理成章就能推導前端如何增加效能了，簡單來說，**減少瀏覽器的「做白工」=== 增加效能**，因此我們可以在 Reflow 與 Repaint 這兩個地方動手腳：

* **Reflow(回流)**：這需要重算 Layout，可以説對效能負擔最重的部分，當因為某些情形，改了元素的寬高、位置，瀏覽器得重新計算整張畫面的佈局。
* **Repaint(重繪)**：改顏色或背景，不影響位置，沒有像回流這樣耗能，但太過頻繁的發生一樣會讓風扇起飛。

既然知道問題發生點，那解法就很簡單了，盡量減少 Reflow 發生的頻率，比如使用 **transform** 或 **opacity** 來處理動畫，而不是用 **top / left** 這種會動到座標、導致佈局需要重算的做法。

或者，使用一種叫做 **Virtual DOM** 的做法，這也是 React 等框架的核心。它的邏輯是：既然操作真實 DOM 很貴（很耗能），那我們就先在記憶體裡算好「最少需要變動的地方」，再一次更新到畫面上，減少瀏覽器做白工的次數。

還記得我前面提到把蛋白粉放到購物車的例子嗎？這裡我們要討論的是「使用者瘋狂點擊」或「輸入框即時搜尋」對效能的負擔，這就涉及 **Debounce（防抖）** 與 **Throttle（節流）** 對效能優化的討論：

* **Debounce (防抖)**： 
  想像你在電梯裡，每進來一個人，電梯門就會重新計時 5 秒才關。如果一直有人進來，電梯就會一直等，這就是防抖的機制。
  用在搜尋框上，因為我們不希望用戶每打一個字就立刻發一次 API，所以我們會設定一個時間，比如 500ms，等他停下來 0.5 秒後，才真正發送請求。
* **Throttle (節流)**： 
  像是在捷運閘門，不管外面排隊的人有多急、刷卡多快，閘門每隔幾秒才放行一個人。
  用在網頁上，比如 **網頁滾動（Scroll）** 的監聽。當要判斷使用者滑到哪裡來觸發 **Lazy Loading（圖片延遲加載）** 時，不需要每 1 像素都計算一次，只要週期性的每 100 毫秒算一次就夠了。

## 要如何直觀檢視到優化的成果？
> 人生也許會背叛你，兄弟會欺瞞你，只有數學不會——數學不會就是不會。

優化不是「感覺變快了」就算數，要是這樣跟老闆或客戶講，怕不是要吃落掛(笑)。在軟體工程的世界，我們有專業的量化工具：

### 1. Lighthouse
![Lighthouse Overview](https://developer.chrome.com/static/docs/lighthouse/images/lighthouse-overview.png?hl=zh-tw)
這款由 Google 開發的開源自動化工具，直接內建在 Chrome DevTools 裡。

**指標說明：**
* **Performance (效能)**： 網頁跑得快不快？圖片有沒有壓縮？
* **Accessibility**： 無障礙功能是否完善？按鈕對比度夠嗎？比如我就有因為按鈕寫 learn more 被扣分，嗯。
* **Best Practices** ： API 是否過期 跟 安全性（HTTPS）是否過關。
* **SEO**： 網頁的關鍵字和 Meta Tag 是否匹配。

**使用指南：**
1. 按下 `F12` 打開開發者工具，切換到 **Lighthouse** 頁籤。
2. 選擇「Mobile」或「Desktop」（建議先測手機版，因為環境更嚴苛）。
3. 點擊 **Analyze page load**。
4. **看報告改錯**： 下方會列出「Opportunities（建議優化項）」，它會直接告訴你哪張圖片太大、哪段 JS 阻塞了渲染。

### 2. Core Web Vitals (核心網頁指標)
![Core Web Vitals](https://www.n7.io/cdn.prod.website-files.com/61779da08a0e587b4ac301a6/678fd21b3f7a4b3362a53ee3_Insight%20Covers%20-%20N7%20(4).png)
這是 Google 用來衡量網站使用者體驗真實感受的核心標準。

**指標說明：**
* **LCP (Largest Contentful Paint)**： 衡量加載性能。網頁最大內容渲染出來的時間，應在 **2.5s** 內。
* **FID (First Input Delay)**： 衡量交互性。用戶第一次點擊到瀏覽器回應的時間，應小於 **100ms**。
* **CLS (Cumulative Layout Shift)**： 衡量視覺穩定性。頁面加載過程中內容跳動的程度，應小於 **0.1**。

## 總結
一個網站的效能如何，絕對不是要將工作丟給後端就可以拍拍屁股丟一邊的。前端的工作其實不比後端輕鬆(大概吧我對後端領域還不夠了解因此先不討論)。

前端需要依靠對 **CRP 關鍵渲染路徑** 的理解，從而避免不必要的 Reflow，再到利用 **Debounce 與 Throttle** 控管事件的觸發頻率，同時要考量使用者的體驗，公司的營收等。雖然我們不用像後端那樣去優化 SQL 語法，但我們守護的是與使用者接觸的第一線——畢竟，一個轉不出來的網頁，內容再好也是枉然。
